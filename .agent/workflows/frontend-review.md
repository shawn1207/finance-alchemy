# 🎨 前端架构师 Code Review 工作流

当你对我发出 `/frontend-review` 指令，或者要求我以资深前端架构师的身份审查特定前端代码或模块时，我将按照以下多个维度和严格标准执行代码审查：

## 🎯 Review 目标
1. 提升代码质量
2. 发现潜在 Bug
3. 优化可维护性和可扩展性
4. 提升性能
5. 符合前端工程最佳实践
6. **保持 UI 代码的一致性和低维护成本**

---

## 🔍 审查维度与标准

### 一、代码正确性
- 是否存在潜在 bug 或边界条件问题。
- 是否存在异常未处理情况或逻辑漏洞。

### 二、代码结构设计
- 模块职责是否单一，组件拆分是否合理。
- 是否存在过大的组件或函数，是否有重复代码可以抽象。
- 是否符合常见设计模式。

### 三、可维护性
- 变量命名是否清晰，函数是否过长，是否存在魔法值。
- 是否缺少必要注释，是否有可读性差的复杂逻辑。

### 四、性能问题
- 是否存在不必要的渲染或重复计算。
- 是否合理使用 memo / useMemo / useCallback（针对 React/前端框架）。
- 是否存在大列表未做虚拟化，是否存在不合理的依赖监听。

### 五、状态管理
- state 是否设计合理，是否存在不必要的 state。
- 是否有 state 可以派生计算，是否存在 props drilling。

### 六、类型安全（TypeScript）
- 类型是否完整，是否存在 `any`，类型是否可以抽象复用。

### 七、安全问题
- 是否存在 XSS 风险，是否有不安全的 HTML 渲染，是否存在敏感数据暴露。

### 八、UI 与样式规范（🚨 重点核心 🚨）
**本项目核心原则：不单独维护 CSS 文件，优先交由 UI 框架或组件库管理样式。**

审查重点：
- ❌ 是否存在独立维护的 CSS / SCSS / LESS 文件。
- ❌ 是否存在大量自定义 className 样式。
- ✅ 是否可以使用组件库自带样式。
- ✅ 是否可以通过组件 props / theme / token 控制样式。
- ✅ 是否可以使用框架推荐方式（如 styled props / utility class / design token）。

**优化原则**：优先使用组件库内置样式能力 > 避免新增 CSS 文件 > 避免复杂样式层级 > 避免样式与业务逻辑耦合。
*如果发现样式问题，审查报告中必须提供推荐实现方式及示例代码。*

### 九、工程规范
- 是否符合 ESLint / Prettier 规范及团队代码风格。

---

## 📝 输出要求与格式

审查报告必须严格遵循以下结构输出：

1. **按严重级别分类**：
   - 🔴 **Critical（必须修改）**：导致 Bug、安全漏洞或严重背离核心规范（如手写大量 CSS）。
   - 🟠 **Important（建议修改）**：性能缺陷、代码结构不合理、可维护性差。
   - 🟢 **Suggestion（优化建议）**：命名规范、类型优化、小范围重构。

2. **每个问题必须包含**：
   - **问题描述**：指出代码片段及所在文件。
   - **原　　因**：解释为什么这是一个问题。
   - **修改建议**：具体的修复指导。
   - **示例代码**：重构或修复后的代码对比。

3. **全局架构建议**：
   - 如果整体设计存在问题，给出推荐的重构方案和示例结构。
