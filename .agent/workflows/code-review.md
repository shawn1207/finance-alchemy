# 🚀 深度代码审查工作流

当你对我发出 `/code-review` 指令，或者要求我审查特定文件时，我将按照以下多智能体角色的逻辑执行任务：

### 1. 架构师角色 (Architect Agent)
**目标**：识别代码在系统架构中的位置，评估是否符合 DDD 设计原则。
- 检查代码是否属于正确的层级 (Infrastructure, Domain, Application, Interface)。
- 检查逻辑是否解耦，是否存在循环依赖。
- 检查接口定义是否契合 `Finance Alchemy` 的整体架构。

### 2. 代码质量专家 (Code Quality Specialist)
**目标**：确保 Python 代码的高质量和可维护性。
- 检查代码风格 (PEP 8)。
- 寻找潜在的性能瓶颈（特别是与 AkShare 或 Eastmoney 通信的部分）。
- 检查异常处理逻辑是否健全（重试机制、错误捕获）。

### 3. 量化审计员 (Quant Auditor)
**目标**：确保金融逻辑的正确性和数据安全。
- 确认财务指标 (PE, ROE等) 的计算公式是否准确。
- 检查并发任务 (Celery/Task) 是否存在竞态条件。
- 验证 API Key 的处理是否安全（不被硬编码在代码中）。

---

## 🛠️ 执行步骤

1. **阅读代码环境**：首先运行 `ls -R` 获取项目结构上下文（如果我还没做的话）。
2. **分析目标文件**：读取你要我审查的特定文件。
3. **分阶段报告**：
   - 给出【架构建议】。
   - 给出【代码优化建议】（包含具体代码重构示例）。
   - 给出【业务安全性提示】。
4. **最终汇总**：给出 1-10 的代码质量评分。
